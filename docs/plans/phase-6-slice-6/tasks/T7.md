# T7 · Per-input filters — universal tier + provider-declared tier

Replaces hardcoded Shorts UI. Two tiers, both per-input, both persisted on the input, applied at
commit.

**Do:**
- **Universal filters** (every input, any provider): at minimum a **title regex** include/exclude.
  Defined once in the generic layer; rendered for every input in the review card.
- **Provider-declared options:** add `App\Providers\InputOption` (key, label, type `bool|enum`,
  default, choices?, applicable input types) + `Provider::inputOptions(ResolvedInput): array` to the
  `Provider` interface. YouTube channel → `include_shorts`, `include_live`; YouTube video/playlist +
  `FakeProvider` → `[]`. Surface both tiers in `PreflightExecutionResult` so the review card renders
  dynamically — no per-provider frontend code.
- Persist chosen values as a JSON `options` column on `stash_inputs` (+ `StashInputRecord` field +
  repository param + migration).
- Apply in `commitInput` discovery → non-matching items → `stash_items.state = ignored` with
  `ignored_reason = filter_title_regex` or `filter_video_type` (retained, not deleted; spec §20).
  Content-type filtering only excludes when `content_type` known (Data API); RSS-only commits skip the
  type filter (title-regex still applies).

**Depends on:** T9 (`commitInput`), T5 (`content_type`).

**Done when:** every input's review shows the universal title-regex filter; a YouTube channel also
shows Shorts/live toggles; excluded items marked ignored with the right reason and retained;
fake/Twitch show no YouTube options.
